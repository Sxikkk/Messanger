# Оценка проекта Messanger

## Краткий вердикт
Проект выглядит как хороший рабочий MVP мессенджера с понятным разделением на слои (`Api`, `Application`, `Domain`, `Infrastructure`, `Persistance`) и уже реализованными ключевыми сценариями: регистрация/логин, refresh/logout, управление сессиями, базовая социальная графика (friends/requests/block), Redis-кэш сессий и PostgreSQL через EF Core.

**Итоговая оценка: 6.5/10 (сильный MVP, но пока не production-ready).**

---

## Сильные стороны

1. **Неплохая многослойная архитектура**
   - Слои разделены по ответственности, есть контракты и сервисы.
   - Доступ к данным вынесен в `Persistance`, кэш — в `Infrastructure`.

2. **Базовая безопасность уже предусмотрена**
   - JWT-аутентификация, refresh token, `HttpOnly` cookie.
   - Пароли хэшируются (BCrypt), refresh token хранится в хэшированном виде.

3. **Нормальная база для масштабирования**
   - Redis и Postgres уже в проекте.
   - Есть `DbContextFactory`, индексы и уникальные ограничения в модели.

4. **Полезный API-контур**
   - Swagger, CORS, middleware, фильтр заголовка устройства.

---

## Основные риски и проблемы

1. **Секреты в репозитории (критично)**
   - В `Api/appsettings.json` находятся JWT key, доступ к БД и SMTP-пароль.
   - Это блокер для безопасного деплоя.

2. **Нет тестов**
   - В репозитории отсутствуют unit/integration/e2e тестовые проекты.
   - Сложно безопасно рефакторить и контролировать регрессии.

3. **Шум и артефакты сборки в репозитории**
   - В Git попали `bin/` и `obj/`, что ухудшает ревью, CI и историю.

4. **Неоднородная обработка ошибок**
   - В контроллерах смешаны `BadRequest`, `throw`, `Console.WriteLine`.
   - Нет единого глобального exception middleware и единого формата ошибок.

5. **Конфигурационные и структурные шероховатости**
   - `Application/Domain/Infrastructure` используют `Microsoft.NET.Sdk.Web`, хотя для части проектов больше подходит `Microsoft.NET.Sdk`.
   - В `Api/Program.cs` `AddControllers()` вызывается дважды (можно упростить регистрацию).

---

## Что сделать в первую очередь (приоритет)

### P0 — обязательно перед публичным запуском
1. Убрать секреты из репозитория:
   - перевести на environment variables / Secret Manager / vault;
   - срочно ротировать уже скомпрометированные ключи/пароли.
2. Включить `.gitignore` и удалить из Git `bin/`, `obj/`.
3. Добавить глобальный middleware для обработки ошибок + единый JSON-формат ошибок.

### P1 — ближайший спринт
1. Добавить тесты:
   - unit для `Application.Services`;
   - integration для auth/session/relations API.
2. Ввести базовую наблюдаемость:
   - структурированное логирование (например, Serilog);
   - корреляция запросов и ключевые бизнес-метрики.
3. Ужесточить CORS и cookie policy для production-окружения.

### P2 — улучшение качества
1. Привести проекты к корректным SDK (`Microsoft.NET.Sdk` там, где не нужен web-host).
2. Очистить дубли и упростить `Program.cs`.
3. Добавить README с архитектурой, запуском и env-переменными.

## Рекомендуемые KPI качества

- Покрытие критичных сценариев тестами: **не ниже 60%** для `Application`.
- MTTR багов auth/session: **< 1 дня**.
- Доля endpoint’ов с типизированными ошибками: **100%**.
- 0 секретов в репозитории по результатам секрет-сканера.

---

## Финальный вывод
Проект уже имеет крепкий фундамент и выглядит как рабочий backend мессенджера на уровне MVP. Главные задачи сейчас — **операционная безопасность**, **тестируемость** и **техническая гигиена репозитория**. После закрытия P0/P1 пунктов проект можно уверенно поднимать до production-пилота.
